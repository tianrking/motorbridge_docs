# MotorBridge 0.2.6 Release Test Note

Date: 2026-05-09
Scope: Rust core/vendor crates, Rust `motor_cli`, C ABI/Python binding, Python CLI. Hardware command coverage is limited to RobStride and Damiao.

This note is meant to be copied forward for every release. It must stay runnable for three install styles: source checkout, release binary, and wheel/PyPI install.

## 0. Safety Legend

- `[SAFE]`: does not move the motor and should not persist device settings.
- `[MOTION]`: may enable or move the motor. Lift the robot, remove load, and keep E-stop ready.
- `[PERSIST]`: writes persistent parameters such as motor ID, zero offset, EEPROM/register config.
- `[DESTRUCTIVE]`: can make the motor harder to find until the new ID/config is known.

For release qualification, all `[SAFE]` checks should pass. Run `[MOTION]` and `[PERSIST]` only on a bench motor prepared for testing.

## Channel Compatibility

The examples below use Linux SocketCAN `can0` because that is the default bench setup for this release note. If the adapter appears as `slcan0`, replace `can0` with `slcan0` in the commands. For CAN bring-up and troubleshooting, see [`docs/en/can_debugging.md`](../docs/en/can_debugging.md) or [`docs/zh/can_debugging.md`](../docs/zh/can_debugging.md).

## 1. Entry Paths

Choose the entry path that matches what is being tested.

### 1.1 Source checkout

Run from the cloned or extracted `motorbridge` repository root:

```bash
export MOTORBRIDGE_SRC="${MOTORBRIDGE_SRC:-$PWD}"
cd "$MOTORBRIDGE_SRC"
cargo build -p motor_cli --release
export MB_RUST_CLI='./target/release/motor_cli'
# For development/debug runs without a separate binary path, this also works:
# export MB_RUST_CLI='cargo run -p motor_cli --release --'
export MOTORBRIDGE_WHEEL='./bindings/python/dist/motorbridge-0.2.6-cp312-cp312-linux_x86_64.whl'
```

### 1.2 Release binary

Run from the directory containing the downloaded release binary, or use a binary already on `PATH`:

```bash
export MB_RUST_CLI='./motor_cli'
# export MB_RUST_CLI='motor_cli'
```

### 1.3 Python wheel or PyPI package

Use a local wheel file, or install the published package after release:

```bash
export MOTORBRIDGE_WHEEL='./motorbridge-0.2.6-cp312-cp312-linux_x86_64.whl'
# After PyPI publish, use: pip install motorbridge==0.2.6
```

Create a local test environment in the current working directory:

```bash
python3 -m venv .venv-motorbridge-0.2.6-test
source .venv-motorbridge-0.2.6-test/bin/activate
pip install --force-reinstall "$MOTORBRIDGE_WHEEL"
# Or after PyPI publish:
# pip install --force-reinstall motorbridge==0.2.6
```

If you are intentionally testing an already installed `motorbridge-cli`, skip the venv and record the package source:

```bash
which motorbridge-cli
python3 -m pip show motorbridge
```

## 11. 0.2.6 Acceptance Criteria

RobStride:

- `motor_id` / `device_id`: explicit `1..255` validation in core, Rust CLI, Python SDK/CLI, and websocket gateway paths.
- `feedback_id` / `host_id`: explicit `0..255` validation in core, Rust CLI, Python SDK/CLI, and websocket gateway paths.
- Rust and Python scan use the exact `--feedback-ids` candidate being tested; scan does not hide an internal host-id fallback.
- Parameter replies are filtered by target `device_id`, reducing cross-motor reply confusion.
- User-facing text says RobStride `feedback_id/host_id` is not the motor ID.
- Python wheel includes new ABI symbols and does not load a stale shared library.

Damiao:

- Rust CLI and Python CLI both cover scan, enable, disable, MIT, pos-vel, velocity, force-position, and ID change flows.
- Damiao SocketCAN and `dm-serial` scan examples are present.
- ID change logs old/new `ESC_ID` and `MST_ID`, then verifies through the new IDs.

Packaging/docs:

- Source checkout, release binary, and wheel/PyPI install paths are documented without machine-specific absolute paths.
- Local Python test environment uses `.venv-motorbridge-0.2.6-test` in the current working directory.
- Canonical Mintlify docs live in the separate `motorbridge-docs` repository; embedded `bindings/python/mintlify` is removed.
